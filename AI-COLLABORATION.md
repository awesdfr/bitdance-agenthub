## 1. 协作的骨架:把「怎么协作」写成文档

本项目最关键的一招:**协作约定本身是版本化的文档**。任何 AI 工具(Claude Code / Cursor / Codex)进来,先读文档,就能在不通读代码的前提下对齐规则与上下文。四类文档各司其职:

```
CLAUDE.md   —— 规则:怎么做 / 不做什么(技术栈锁定、五层架构、安全红线、协作纪律)
OVERVIEW.md —— 地图:做了什么 / 代码在哪(功能矩阵 + 文件定位)
specs/      —— 规格:每个模块的字段与契约(14 篇)
skills/     —— 配方:几类"会反复做"的扩展任务的步骤化指南(4 篇)
```

这套结构带来的直接效果,在 file2 开场就能看到 —— 一个**全新会话**仅凭读文档 + 读代码就接续了前一天中断的工作(详见 §2 支柱四)。

> 💡 这正是 AgentHub 自身的产品理念在开发流程里的自我印证:它做的是「多 Agent 协作平台」,而它自己就是用「文档驱动的人机协作」造出来的。

## 2. 方法论四支柱(原则 + 实录佐证)

### 支柱一 · 先读规范,再动手

**原则**:进入一个有约定的代码库,第一动作是读约定,而不是凭习惯开干。规则优先于个人风格。

**实录**:克隆完项目、被要求装依赖时,Claude 的第一步是加载 `CLAUDE.md`、提取包管理器规则,并和 lockfile 交叉验证 —— 哪怕环境里 `npm` 已经可用,也坚持用项目规定的 pnpm:

> **file1:23-26**
> ```
> ⎿  Loaded bitdance-agenthub\CLAUDE.md
> ● 项目要求使用 pnpm（CLAUDE.md 明确规定，且有 pnpm-lock.yaml）。先检查 pnpm 是否已安装。
> ```

更进一步:遇到一个有现成第三方包能解的技术问题,Claude **主动不引入依赖**,并援引具体条款,还把「为什么不引入」写进 spec:

> **file1:1856-1858**
> ```
> **Why 不读 Windows hidden attribute**：Node fs.statSync 不暴露此 attribute；
> 引入第三方包（如 winattr）违反 CLAUDE.md §4.3「不为将来可能用到加抽象」，
> 且这些已知名能覆盖 95% 噪音目录。
> ```

这是「遵守规范」最难得的一种:**手里有现成的快解法,却因为一条规则而克制不用**,并留下可追溯的理由。同类自我克制还出现在拒绝引入 `iconv-lite`(援引 §6.2「加依赖先讨论」)。

---

### 支柱二 · Spec 驱动:文档与代码同呼吸

**原则**:接到「实现 X」,先确认/编写 spec,让人过目,再写代码;实现推进时,spec 与代码**同步更新**,绝不让规格沦为过期的一面之词。

**实录 1 —— 大特性先分阶段**:整个「跨 run 对话记忆」被切成 Phase A–E,「从最小可用到完备」,A 标注「最关键,先做这个」(file1:6272-6317)。

**实录 2 —— 先写 spec,过目,再实现**:

> **file1:6568-6605**
> ```
> ● Task #21：写 spec 13。
> ● Write(specs\13-conversation-context.md) ⎿ Wrote 230 lines
> ● Spec 13 写完。给你过目一下要点，然后进入实现：
>   ...
>   9. 群聊场景 Phase C 才做，spec 已把契约定下来了
>   继续实现 Task #22-#25。
> ```

编号任务里,「写 spec」本身就是一个 Task,且排在实现 Task 之前 —— spec 不是实现的副产品,是实现的前置。

**实录 3 —— spec 与代码同步迁移**:当 Phase D 真正实现 token 预算时,spec 里那条「故意不做的事」被**物理地迁进正文**,从「待办」变成「已规格化的算法」:

> **file1:7407-7418**
> ```
> ● Task #33：更新 spec 13 把 token 预算从「故意不做的事」移入正文：
> ● Update(specs\13-conversation-context.md) ⎿ Added 52 lines, removed 3 lines
>      201 -## 故意不做的事
>      201 +## Token 预算
> ```

**实录 4 —— 续接时先读 spec、并校正设计漂移**:Phase C 落地时,spec 标题从「本 spec 现在不实现,但提前定好契约」改为「已实现」,同时**把实现与原设计的分歧写回 spec**(早先设想的 `policy` 开关被「按 agentIds 数自动判定」取代),保证 spec 是代码的忠实镜像而非陈旧计划(file2:223-268)。

### 支柱三 · Claude 是思考伙伴,不是应声虫

**原则**:好的协作不是「让做什么就做什么」。AI 要能指出需求里的反模式、提出更优方案、把 trade-off 摆上台面,必要时**有理有据地说"不建议"** —— 同时永远把最终决定权留给人。

**实录(旗舰)—— 否决「省流/全量」开关**:用户提议加一个省 token / 费 token 的二选一开关。Claude 没有顺手照做,而是**否决**并给出三条理由,再提出替代方案,最后还留了一个「你若坚持,我能接受的最小形态」的台阶:

> **用户(file1:6926-6928)**
> > 我想再问一下可以加一个更用户侧体验的说明按钮（比如省流模式和全量模式）…这种可以加吗，你觉得呢
>
> **Claude(file1:6939-6950)**
> > 说实话不太建议——「省流 / 全量」二选一是典型的「看起来贴心实际上是把决策推给用户」的设计。理由有三：
> > 1. **真实需求是一个滑动条而不是开关**:影响 token 的有 maxTurns × includeThinking × groupChatSharing × workspaceContextSize 至少 4 个维度。二选一逼用户接受一组绑定…两边都骂。
> > 2. **用户没有判断能力做这个选择**…决定要看的是「我现在用了多少 token、上下文窗口还剩多少」——这是信息显示问题,不是模式切换问题。
> > 3. **跟项目现有风格冲突**:项目到现在都是 spec-driven 默认值…加这一个会开「让用户调」的口子。

它否决的不是「做不做」,而是「这个设计对不对」,且每条理由都落在项目自身的设计哲学上 —— 这是思考伙伴而非执行器的标志。

**实录 —— 两次劝退 Phase E,并重定优先级**:被问「Phase E 有没有必要做」,Claude 先指出原计划忽略的技术风险(resume 失败需要回退兜底),用户再追问时给出明确结论,并把优先级**掉头**指向一个它早先 flag 过、ROI 更高的真问题:

> **file2:1348-1374(节选)**
> > 我直接给结论:**不值得现在做**。…有个明显更值的事在等:我之前 flag 的 sub-agent 双份上下文…**这是每一次派发都在白烧 token,现在就在发生**,影响面比 Phase E 大得多。…如果你说"我就是天天用 ClaudeCode",那我立刻把 Phase E 拎回来…否则我建议掉头去做去重那个。**你定方向。**

用户随后采纳了 Claude 重定的优先级。**有立场,但留出口**。

**实录 —— 动 git 历史前先解释、再征求同意**:用户对「打成单独 commit」不确定,Claude 没有直接跑 git,而是讲清取舍、列出「不会做的事」(不 push、不动 main)、点出工作树里改动混在一起的复杂性,然后问「要我现在分 3 个 commit 吗?还是你自己提?」(file1:2828-2864)。对不可逆操作,先确认再动手。

### 支柱四 · 验证优先 · 根因修复 · 跨会话续接

**原则**:不轻信(包括不轻信用户的 bug 描述和自己上一步的结论);改 bug 找根因不打补丁;宣布完成前用真实工具输出自证;跨会话时先核实状态再继续。

**实录 —— 跨会话续接先验证状态**:file2 是一个全新会话,接续 file1 的中断点。它没有盲信旧日志,而是「先读中断的对话 → 独立核实代码现状 → 确认上一个任务确实在位 → 读 spec 再写下一个任务」:

> **file2:32-45(节选)**
> ```
> ● …Let me verify the current state of the code and read the relevant files.
> ● Task #35 (conversation-context.ts) is fully in place. Now I need to continue Task #36…
> ● Task #35 is complete. Now let me read the spec to ensure Task #36 follows the intended design and to keep it in sync.
> ```

**实录(旗舰)—— 三 bug 根因表 + 端到端验证**:修 Electron 打包的漏拷问题,Claude 不是逐个打补丁,而是把每个 bug 拆成 **Bug → 根因 → 修复**,并以一次真实的 `HTTP 200` 收尾:

> **file1:6175-6200(节选)**
> ```
> ● 完全跑通——无报错、HTTP 200、API 正常。
>   修复总结  prebuild 脚本三个 bug 一次性修掉：
>   Bug: @swc/helpers 等漏拷
>   根因: findInPnpm 选中了 peer-dep 变种里的 symlink，cpSync 没 dereference…
>   修复: findInPnpm 优先返回 canonical 真目录；cpSync 加 dereference: true
>   ────
>   Bug: 跨 react/react-dom 这种「Windows 不可跟随」的 SYMLINKD
>   根因: next build 留下的相对 symlink 在 Windows 上 statSync 返回 EPERM…
>   ...
>   - 端到端验证过：装到任何 Windows 机器都不再需要手动补依赖
> ```

**实录 —— 坦承自己上一个补丁才是 bug**:更难得的是,Claude 把**自己先前的修复**点名为根因之一,并解释为何会崩、正确做法是什么:

> **file1:5036-5040(节选)**
> > 我之前那个 prebuild 补丁踩坑了:我把 …这种带哈希后缀的 symlinks 当成「7-zip 不认的废链接」删掉了;实际上 Next.js 编译后的代码里有 `require('better-sqlite3-cf218e5bd1d5f04c')` 这种调用…删掉就直接 runtime crash。正确做法是把那些 symlinks 替换成真目录。

**实录 —— 宣布完成前用真实输出自证**:`pnpm typecheck` 通过、`git diff --stat` 核对改动面,并注意到 diff 里还带着「上一个会话未提交的 Task #35」(file2:169-401);lint 时只检查自己动过的文件,证明唯一的 warning 是既有噪音、与本次改动无关(file2:374-385)。**「干净」是用工具输出证出来的,不是声称的。**

## 3. 人机分工

这套协作里,人和 AI 的边界很清楚:

| 角色 | 负责 |
|---|---|
| **人(项目负责人)** | 定方向、拍板 trade-off、提供领域判断(「我天天用不用 ClaudeCode」)、对不可逆操作授权、踩刹车 |
| **Claude** | 读规范上手、写/同步 spec、分解任务(Task #NN)、实现、根因排障、自检自证、**坦诚报告失败**、**有据地否决坏设计**、跨会话续接 |

关键不在「AI 干了多少」,而在**边界清晰**:Claude 主动承担「分析、提议、自检、说不」,但「方向与拍板」始终交还给人 —— 上面每个「你定方向」「要我…吗?」都是这条边界的实证。

## 4. 纪律红线(从实录提炼)

这些不是墙上的口号,是实录里**真正被执行**的约束:

- **加依赖先讨论**(CLAUDE.md §6.2)—— 两次主动不引入(`winattr` / `iconv-lite`),见支柱一。
- **一个 commit 一件事**(§7.2)—— 为此把跨 Phase 的单个文件拆成分块暂存(file1:3093),宁可麻烦。
- **改 spec 接口先停下对齐** —— 涉及 DB schema 变更时进入 plan 模式先要签字(file2:1006-1021)。
- **所有 LLM 调用带 AbortSignal / 跨进程输入过 zod / fs·bash 过沙箱** —— bash 工具的中止从「单进程 SIGTERM」升级为「平台感知杀进程树」(file1:921-998)。
- **不主动 push** —— 「按 CLAUDE.md 默认,不会主动 git push。你想推时再说。」(file1:4327)

## 5. 可复制清单:如何在本项目与 AI 协作

把上面的方法论压缩成一份操作清单 —— 下一个 AI 会话(或新同学)照着做即可复现这套协作质量:

1. **先读后做**:开工前读 `OVERVIEW.md`(地图)+ `CLAUDE.md`(规则);要扩展某子系统先看 `skills/` 对应配方。
2. **Spec 先行**:接到「实现 X」,先确认/写 `specs/` 对应规格并过目,再写代码;实现时 spec 与代码同步改。
3. **小步 + 单一职责**:一个 commit 一件事;100 行能解决别写 500 行。
4. **该停就停**:加依赖、改 spec 接口、删多处引用的符号、动安全约束 —— 先问,不自作主张。
5. **思考而非服从**:需求有反模式就指出来,给替代方案,但把拍板留给人。
6. **验证优先**:改前先核实状态,改 bug 找根因,宣布完成前 `pnpm typecheck` / `pnpm lint` + 看 diff 自证。
7. **诚实报告**:失败就说失败(包括自己造成的),不含糊其辞。
